hpc-bridge — how it works

A Claude Code plugin that lets an agent stand up real HPC compute, run on it, and tear it down — through four tools and a credential-free hot path.

Current as of the Anvil (Purdue/ACCESS) integration · branch claude-effort
The one idea: there are two separate channels, and the split is the whole design. SSH is the control channel — used only to stand up / tear down the endpoint (a couple of times per session). Globus Compute is the work channel — used for every command, over a Globus Auth token, never SSH. So the tool Claude drives constantly (run_shell) never touches your credentials.

Lifecycle of a request

ensure_endpoint_up — stand it up

  1. Server already chose SlurmFacility (from HPC_BRIDGE_FACILITY=anvil).
  2. Over SSH: configure (if new) → write the manager + engine config → start --detach.
  3. The manager registers with Globus and gets a UUID.
  4. Poll status via the SDK → return {status:up, endpoint_id}.

run_shell("nproc") — do work

  1. Wrap the command in the session shim, submit a ShellFunction via a long-lived Executor (AMQP, Globus token).
  2. First call: the manager asks Slurm for a node — the ~40–70 s cold start. The worker runs it.
  3. Result → ShellOutcome (exit_code, stdout, cwd, session_spend). Later calls hit the warm worker (~1–2 s).

Why it feels like a REPL

Globus runs each command in a fresh subprocess on the worker, so cd/export wouldn't survive between calls. The server invisibly wraps every command to rehydrate, then persist the working directory + environment in small files on Anvil's Lustre scratch:

cd "$(saved cwd)"; source "$(saved env)"; <your command>;  then save the new cwd + env

So cd build in one turn and make in the next "just work" — Claude sees a persistent shell with zero discipline required.

Warm vs. cold (and the cost knob)

  • interactive → Slurm min_blocks=1: the node is held warm for fast back-and-forth — but the allocation is charged the whole time. That's why ending with stop_endpoint matters.
  • batch (default) → min_blocks=0: scales to zero between commands — cheap, but each command pays a cold start.
  • Every result carries block_state + session_spend, so warmth and cost are visible to Claude without a separate tool.
  • The endpoint is a v4 Manager Endpoint Process: the manager lives on the login node; a worker (User Endpoint Process) is spawned on a compute node only when the first task arrives.

How it's wired — and how it generalizes

  • The repo is the plugin (.claude-plugin/plugin.json + .mcp.json). .mcp.json launches the server over stdio with uv run --directory <plugin> --extra integration hpc-bridge (uv run so it always builds current source).
  • Config is environment: HPC_BRIDGE_FACILITY, SSH user/key, and the Slurm account tell the server which machine and how to reach it.
  • Machine specifics are data: everything about Anvil lives in one MachineProfile (account, ib0, debug partition, the worker's module load + venv). The lifecycle / dispatch / session / cost code is facility-agnostic, behind one Facility seam — so NERSC, Midway, DeltaAI, … are each just another profile.

What's built vs. designed

BUILT

  • Agent-driven provision → run → teardown on a real Slurm facility (Anvil), over key-based SSH + Globus Compute.
  • Session-persistent shell on the shared filesystem; cost-gated warm/batch profiles; structured, never-crashing tool results.
  • Profile-driven facility seam; cross-platform plugin packaging; BYO-endpoint mode.

DESIGNED, NOT YET BUILT

  • Separate-UID credential broker + MFA-over-localhost for OTP facilities (NERSC/ALCF/OLCF/TACC). Anvil's key-based SSH let us skip it.
  • Durable task handles (recover results across restart); automatic idle-down; per-job resource requests (GPUs / N nodes / walltime).

The honest boundary

The SSH key lives on your Mac and is used only for bootstrap — run_shell executes on the Anvil worker, so it can't read your Mac's key. But it does run as you on Anvil, so it can read your Anvil-side files and tokens. That irreducible "the agent runs as you" exposure is the same as handing an agent your shell, and it's the conversation to have with facility security before any broader deployment.

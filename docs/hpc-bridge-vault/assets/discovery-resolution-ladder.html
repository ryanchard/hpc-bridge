The resolution ladder — inside connect_facility()

The agent only ever calls two discovery tools. Everything else — the cache read, the index query, the SSH probe — happens server-side, inside a single tool call, walked top-to-bottom until something resolves. The agent never touches facilities.json.

what the agent calls · visible in the trace

list_facilities(query)

Browse the curated Globus Search index. Returns agent-safe summaries. The explicit “what machines exist?” step. :645

connect_facility(facility, ssh_host?, details?)

Resolve and bind. This is the only call that walks the ladder below — cache, index, and probe all live in here. :718

Local discovery is embedded, not a step. For a catalogued machine you usually see list_facilities then connect_facility. For a cached machine you see only connect_facility(ssh_host=…) — the cache hit is invisible; the tell is an immediate needs_account with no proposed_facility_details and no SSH.

deviations · where the agent breaks the flow

bypass

Skips connect_facility

Reasons about SSH/Duo or reaches for login_shell before connect_facility. The cache lives inside that call, so it's never consulted → needless re-probe or “please SSH in”.

key miss

Inconsistent ssh_host

The cache key is the raw string. Confirm with midway3.rcc.uchicago.edu, reconnect with midway3 → different key → miss → re-probe. (Endpoint name sanitizes; the key doesn't.)

override

Fabricates details

Invents a details= / endpoint_name mid-flail (the recover-00 we saw) — an explicit details= overrides the ladder, writing the wrong config to cache.
